# SPEC-SCHEMA

## Purpose

This file is the authoritative schema for `lab/runtime/<round_id>/spec.json`, the per-round manager manifest introduced in R7 G-A. The manager authors spec.json before dispatch so that worker adapters and the human reviewer share a single source of truth for that round's scope. Workers cite spec.json in their final reports; the evaluator (`evaluate-spec.ps1`, landed in R9) consumes it to emit verify.json; and future regression rounds validate it against the rule catalog defined here. Any field used by a rule or by an adapter MUST be described in this document, and any extension MUST bump `spec_version`.

## Required Fields

Each top-level field of spec.json is described below: shape, purpose, and example value. Field order in the JSON file is not significant, but every required field MUST be present.

- `spec_version` (string, semver-draft) — schema version of this spec.json. The evaluator dispatches its parser by major version. Example: `"0.3.0"`.
- `round_id` (string) — short round identifier matching the runtime subdirectory. Example: `"R9"`.
- `round_group` (string) — one of `G-A`, `G-B`, `G-C`, `G-D`, or a future group label. Drives reviewer expectations and dashboards. Example: `"G-C"`.
- `title` (string) — human-readable round title; appears in manifest.json and reports. Example: `"Architect the spec.json schema"`.
- `objective` (string, multi-sentence) — prose statement of what this round must accomplish and why. Used as the canonical brief; workers SHOULD echo it back. Example: `"Author SPEC-SCHEMA.md so that the evaluator and future regression rounds share a stable contract..."`.
- `inputs[]` (array of absolute path strings) — files the round depends on for context. Typically includes manager-temp prompt files, prior round verify.json, and any reference docs. Example: `["C:\\Users\\darkh\\Projects\\.deskrelay-manager-temp\\.tmp-r9-architect-prompt.txt"]`.
- `assignments[]` (array of objects, one per worker) — each entry has:
  - `role` (string) — e.g. `"architect"`, `"implementer"`, `"reviewer"`.
  - `adapter` (string) — adapter name that will execute the role (e.g. `"claude-oneshot"`).
  - `allowed_paths[]` (array of absolute path strings) — write allowlist for that worker.
  - `forbidden_paths[]` (array, optional) — explicit denylist when allowed_paths is broad.
  - `expected_artifacts[]` (array, optional) — files the worker is expected to produce; consumed by `exists` checks.
  - `expected_duration_ms` (optional number) — manager's pre-dispatch estimate. The evaluator flags any assignment whose actual durationMs exceeds 1.5x this value as a `workload-imbalance` row in FAILURES.md (layer=protocol). R24 measured parser.ts at 140s vs cli.ts at 14s in the same round — 10x imbalance unflagged. Future rounds with this field can split or merge work pre-dispatch.
- `declared_checks[]` (array) — rule library entries the evaluator will run. See [Rule Catalog](#rule-catalog) for supported rule types.
- `facts_sources[]` (array of filenames) — files whose content is authoritative for this round; reviewer disagreements MUST be resolved against these. Example: `["SPEC-SCHEMA.md", "lab/runtime/R9/spec.json"]`.
- `derived_views[]` (array of filenames) — files computed from facts (e.g. dashboards, summaries). These are manager-only outputs and MUST NOT be directly edited by workers.
- `defeater_fields` (object) — keys are `check_id_or_concern` strings; values are prose describing how that check or concern could be wrong or insufficient. This is the round's self-skepticism record; it is prose-only and not machine-checkable.
- `canonical_examples[]` (optional, REQUIRED when assignments.length > 1 AND any assignment uses prose-described formats per PROTOCOL.md ## Worker Prompt Schema field 8) — list of objects with `{name, purpose, content}` providing literal byte-for-byte artifact examples that disambiguate spec prose.
- `module_export_contract[]` (optional, REQUIRED when the round dispatches more than one implementer AND any implementer module is imported by another via `import { X } from './path'` syntax) — list of `{ module, exports[] }` objects enumerating the public export symbol names per module. The R53 lint `export-contract-match` (see ## Rule Catalog) verifies that every `import { X } from './module'` substring in the worker's prompt is satisfied by `module_export_contract.find(m=>m.module===module).exports.includes(X)`. F-R47-1 incident (R47 ts-rename round): implementer-stages worker imported `writeAtomic` from `./checkpoint` but implementer-fs worker only exported `writeCheckpoint`. The R47 architect HAD named "import name drift" as a defeater, but pre-emptive naming did not prevent the actual drift. With module_export_contract declared up front and lint-enforced, the spec MUST enumerate `writeCheckpoint`, `writeAtomic`, etc., and the lint fails the dispatch when the enumeration omits an imported symbol.
- `runtime_verification` (object, REQUIRED when the round's deliverable is runnable — has a build output, is a script the user invokes, or has a prior-round smoke harness) — declares the end-to-end runtime smoke step the manager runs at round close. Shape: `{ "command": <string>, "expected_exit": <int>, "expected_stdout_substrings": [<string>...], "expected_files": [<string>...], "headless_browser_shim": <string, optional>, "real_browser_required": <bool, defaults true when the deliverable renders to DOM/canvas>, "interactions": [ { "key": <string, KeyboardEvent.code>, "hold_ms": <int>, "assert_state_predicate": <string, JS expression evaluated against the page's window after the key is released> } ], "forbid_shim_self_output": <bool, defaults true>, "os_matrix": <object, optional, REQUIRED when the deliverable relies on POSIX signals / fork-exec / case-sensitive filesystem; shape { "supported": [<os>...], "known_skips": { <os>: [<assertion_id>...] } } where os values are 'linux' | 'darwin' | 'win32' and assertion_id values are author-chosen strings the shim uses to identify which assertions to skip per OS at runtime via process.platform>, "expected_additions": [<path>...] }`. The command MUST drive the artifact the way the end user would and MUST NOT re-run the worker's own self-tests. When `real_browser_required` is true the shim MUST use a real browser engine (chrome.exe --headless, playwright, puppeteer) — `vm.runInContext` style mocked-DOM evaluation is rejected (F-R34-1). When `interactions[]` is non-empty the shim MUST drive each key through the real browser and evaluate the predicate against window-scoped state; a keyboard-driven artifact with an empty interactions[] is a layer=verification failure (F-R34-2). When `forbid_shim_self_output` is true the round_open evaluator scans the shim source and rejects any `process.stdout.write` of a literal matching `expected_stdout_substrings[*]` — preventing the F-R34-1 anti-pattern where the shim wrote the substring it then asserted. R30 incident (Tetris page-dead-on-first-open) + F-R34-1 (file://+module CORS missed by mocked-DOM shim) + F-R34-2 (modeSelected gate blocked Space-only start, no keypress in shim) are all closed by this field's R35 extension. os_matrix declares which operating systems the verifier is expected to pass on and which assertions are known to be skipped per OS. F-R41-1 (R41 memo-service): the smoke shim asserted the child emitted 'graceful shutdown complete' after SIGINT, but Bun on Windows does not deliver POSIX signals to child processes spawned via Bun.spawn — the marker never appeared on Windows. With os_matrix.known_skips.win32 declaring the skip up front, the shim gates the assertion on process.platform without needing a manager direct edit. expected_additions declares files the round's own implementer workers will add to the lab/workspace tree; the round_close smoke compare for hash-diff-set / regenerate-views-style inventories MUST mask the inventory keys these additions affect. F-R44-1 incident: the R44 lib-common refactor itself added lib-common.ps1 + smoke.ps1 to the lab tree; the byte-identical compare for regenerate-views.json's dry-run summary failed not because of refactor regression but because the inventory counted 2 new files. R45 fix landed this as a smoke.ps1-internal allow-list (`$expected_additions = @('lib-common.ps1','smoke.ps1')`); R53 promotes it to a SPEC-SCHEMA field so any future refactor-class round can declare its own additions without smoke.ps1 surgery.
- `violation_budget` (object) — `{ "manager_direct_edits_max": <int>, "current": <int> }`. Tracks how many direct manager edits to worker-owned files are permitted in this round and how many have been spent.
- `round_close_criteria` (string) — prose statement of what must be true for the round to close cleanly (e.g. `"verify.json shows zero FAIL rows and manifest.json status=closed."`).

## Rule Catalog

The evaluator (`evaluate-spec.ps1`) supports the following rule types in `declared_checks`. Each entry MUST include `id`; other parameters are rule-specific. Pass condition is stated below the parameters.

- `exists` — `{ path }`. Pass when the file or directory at `path` exists on disk after the round's writes complete.
- `select-string-presence` — `{ file, pattern OR patterns[], must_match | must_match_gte | must_match_each_gte | must_match_each }`. Implemented with `Select-String -SimpleMatch`. Pass when the chosen quantifier is satisfied: `must_match` is an exact count, `must_match_gte` is a floor, `must_match_each` and `must_match_each_gte` apply per-pattern when `patterns[]` is used.
- `preserved-headings` — `{ file, headings[] }`. Pass when every heading in `headings[]` appears in `file` exactly once, in any order.
- `line-cap` — `{ file, max_lines }`. Pass when `file` has at most `max_lines` lines (newline-terminated count).
- `hash-diff-set` — `{ expected_changed[], expected_added[], expected_removed[]?, expected_unchanged_gte }`. Compared against `pre-snapshot.json` taken before dispatch. Pass when the actual diff set matches the expected sets exactly and the unchanged count meets or exceeds `expected_unchanged_gte`.
- `v08-equivalent` — `{ scope, expected_adds[] }`. Pass when no unexpected top-level files have been added under `scope`; only the entries in `expected_adds[]` may appear as new top-level paths.
- `subdir-add` (V09) — `{ expected_new_dirs[], expected_subdir_files{} }`. Pass when only the listed new subdirectories appear and each contains exactly the files declared in `expected_subdir_files`. Disallows stray subdirectories.
- `runtime-smoke` — `{ command, expected_exit, expected_stdout_substrings[], expected_files[], headless_browser_shim? }`. Drives the round's deliverable end-to-end. Pass when the command exits with `expected_exit` AND every `expected_stdout_substrings` literal appears in captured stdout AND every path in `expected_files` exists after the run. The rule fires whenever spec.json declares a `runtime_verification` block; if the round declares the block but no `declared_checks` entry references it, that itself is a `layer=verification` failure.
- `path-consistency` — `{ html_files[], build_output_dir, expected_script_pattern }`. For each HTML file in `html_files`, scans for `<script src="…">` and asserts every referenced relative path resolves to a file currently on disk under the round's lab/workspace root (or under `build_output_dir` if specified). Catches the R30 incident class: index.html referencing ./main.js while the actual built artifact is ./build/main.js. Pass when every script-src reference resolves to an existing file.
- `runtime-interaction-smoke` — `{ command, interactions[], assertion_timeout_ms? }`. Drives the deliverable through a real browser engine, presses each key in `interactions[]` (holding for the entry's `hold_ms`), then evaluates the entry's `assert_state_predicate` against the page's `window` after the key is released. Pass when the shim exits 0 AND every predicate evaluates truthy within `assertion_timeout_ms` (default 2000). The rule fires whenever spec.json declares `runtime_verification.interactions` with at least one entry; a keyboard-driven deliverable that declares no interactions is itself a `layer=verification` failure. F-R34-2 incident: pikachu-volleyball Stage 3 shim did `vm.runInContext` of main.js and asserted 3 boot substrings but never pressed Space, so the `modeSelected` start-gate defect that blocked all user-facing play was invisible to the rule.
- `os-matrix-coverage` — `{ os_matrix, current_platform }`. At round close the evaluator reads the spec's runtime_verification.os_matrix.supported list and asserts that the current_platform (the OS the manager ran on) is in supported; otherwise the round records a workspace-divergence warning. Additionally, for each known_skips entry the evaluator notes that the corresponding assertion will be conditionally skipped at runtime — this is documentation, not a failure. The rule fires whenever spec.json declares runtime_verification.os_matrix; if os_matrix is missing entirely on a runnable artifact that uses POSIX signals (heuristic: shim source contains `process.on('SIG` or `kill('SIG`), the rule emits a `layer=workspace` warning rather than a hard fail (since older specs predate R42). F-R41-1 incident motivates this rule.
- `export-contract-match` — `{ module_export_contract, worker_prompts }`. At round close (or at validate-spec lint time pre-dispatch), parse every worker prompt for substrings matching the pattern `import\s*\{\s*([^}]+)\s*\}\s*from\s*['"]\.\/([^'"]+)['"]` (TypeScript named imports from a relative module path). For each captured `{ symbols }` and `module`, check that `module_export_contract.find(m => m.module === module + '.ts')?.exports` contains every symbol. Fail with `unsatisfied_imports[]` listing the offending `module → symbols` pairs.
- `duration-budget` — `{ assignment_role, expected_ms, observed_ms, tolerance }`. Pass when `observed_ms` ≤ `expected_ms` × `tolerance` (default tolerance 1.5). Fail emits a `workload-imbalance` finding.

## Lifecycle

- Manager takes `pre-snapshot.json` of repo state before dispatch.
- Manager authors `spec.json` for the round, including `declared_checks` and `assignments`.
- Manager opens `manifest.json` with status=open and links to spec.json.
- Manager dispatches each worker per its assignment; adapters write to `audit.log`.
- Workers produce their `expected_artifacts`.
- Manager runs `evaluate-spec.ps1` against `spec.json`.
- Evaluator emits `verify.json` with per-check PASS/FAIL rows.
- Manager performs bookkeeping (defeater review, violation_budget update, derived_views regeneration).
- Manager closes `manifest.json` with status=closed once `round_close_criteria` is met.

## Versioning

The `spec_version` field allows future schema migrations without breaking historical rounds. The evaluator selects its parser variant by major version: a `1.x` evaluator MUST refuse a `2.x` spec.json and vice versa. Minor versions are backward-compatible additions only (new optional fields, new rule types); patch versions are clarifications. Removing or renaming a required field MUST bump the major version.

## Workload Balance

The goal of workload balance is dispatch_time-detection of poorly-sized work rather than post-mortem analysis after the round closes. R24 demonstrated that a 10x duration spread between sibling assignments (parser.ts 140s vs cli.ts 14s) can go entirely unflagged when balance is only assessed retrospectively, so the manager MUST estimate and the evaluator MUST gate at dispatch boundaries.

Manager checklist:
- Estimate every assignment's `expected_duration_ms` before dispatch.
- Any single assignment estimated > 2x the median across the round's assignments is split or absorbed into another.
- If any actual run exceeds 1.5x its estimate, file a `workload-imbalance` failure via the `duration-budget` rule.

## Defeaters

- Some rule types may be inadequate for content-level checks: `select-string-presence` and `preserved-headings` verify surface tokens, not semantic correctness, so a worker could satisfy every declared_check while still producing wrong content.
- `defeater_fields` is prose-only; it captures the manager's stated concerns but is not machine-checkable, so a missing or perfunctory entry will pass evaluation even though it weakens review quality.
- `spec_version` migration policy is documented here but not yet enforced by `evaluate-spec.ps1`; until the evaluator gates on major version, a mismatched spec.json may be parsed with the wrong rules and silently produce misleading verify.json output.
- The rule catalog does not yet cover cross-file invariants (e.g. "every facts_sources entry must be referenced by at least one assignment"), so structurally inconsistent spec.json files can still pass.
- `hash-diff-set` depends on `pre-snapshot.json` being taken at the right moment; if the manager mis-orders snapshot vs. dispatch, the diff set is meaningless even when the rule reports PASS.
