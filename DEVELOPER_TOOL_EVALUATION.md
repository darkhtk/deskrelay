# DeskRelay 개발도구 평가

평가일: 2026-05-11  
대상: `self` 프로젝트, `main` 브랜치 기준  
전제: 일반 소비자 앱이 아니라, 파워유저가 자기 PC와 네트워크 안에서 직접 운영하는 self-host 개발도구로 평가한다.

## 결론

DeskRelay는 아이디어와 기능 밀도만 보면 강한 개발도구다. 브라우저에서 여러 PC의 Claude Code 세션, 권한, 스킬, 지침, 작업 폴더, 이미지, 사용량을 다루는 방향은 분명한 차별점이 있다.

하지만 개발도구로 냉정히 보면 아직 "믿고 설치해서 계속 쓰는 도구"라기보다는 "구조를 이해하는 사람이 고쳐가며 쓰는 강한 실험 도구"에 가깝다. 가장 큰 약점은 기능 부족이 아니라 설치, 등록, 네트워크, 재부팅 복구, 실제 원격 PC 검증의 신뢰도다.

## 강점

1. 기능 방향이 분명하다.

   "내 PC에 있는 Claude Code를 브라우저로 제어한다"는 설명은 강하다. 일반 채팅 앱이 아니라 원격 개발 control plane이라는 정체성이 있다.

2. self-host 철학이 설득력 있다.

   서버가 사용자 PC에서 돌고, 디바이스 등록 상태와 토큰도 로컬에 남는다. 출시용 SaaS를 포기한 지금 방향과 잘 맞는다.

3. 파워유저 기능이 많다.

   세션 목록, 폴더별 세션 삭제, 권한 편집, 스킬 목록, 지침 관리, 이미지 렌더링, 사용량 표시, 업데이트 기능은 단순 remote shell보다 훨씬 강하다.

4. 테스트가 없는 프로젝트는 아니다.

   `site-frontend` 테스트가 많고, self-host virtual/failure smoke 스크립트도 있다. 적어도 UI와 일부 설치 로직은 회귀 방지 의식이 있다.

5. 구조 설명이 문서화되어 있다.

   README의 Mermaid 구조도와 등록 흐름은 파워유저에게 도움이 된다.

## 약점

1. 설치가 아직 도구의 신뢰를 갉아먹는다.

   DeskRelay의 가치는 설치 후에야 드러나는데, 설치 전 단계에서 Git, Bun, PowerShell execution policy, PATH, Tailscale, firewall, port conflict가 모두 실패점이다.

2. "등록됨"과 "사용 가능함"의 차이가 여전히 크다.

   device row가 생겼다고 끝이 아니다. 서버가 daemon에 접근 가능해야 하고, daemon token이 맞아야 하고, behavior가 떠야 하고, Claude Code가 실행 가능해야 하고, workspace root도 맞아야 한다.

3. 실제 다른 Windows PC 검증이 부족하다.

   문서상 가장 큰 남은 결함도 이 지점이다. 가상 smoke와 단위 테스트가 있어도 실제 사용자 PC의 꼬인 상태를 충분히 대변하지 못한다.

4. 상태 진단 모델이 아직 완성되지 않았다.

   UI, 로그, command 파일, scripts, diagnostics가 각각 정보를 갖고 있다. 하지만 하나의 doctor 결과로 "지금 이 노드가 실패했다"를 단정하는 계기판은 부족하다.

5. 보안 모델이 파워유저에게 많이 의존한다.

   공용 인터넷에 connector port를 노출하지 말라는 전제는 맞다. 하지만 토큰이 URL과 command에 들어가고, 원격 PC에서 Claude Code를 실행하는 도구인 만큼 경고와 제한이 더 강해야 한다.

6. UI는 강하지만 초반 인지 부하가 크다.

   세션, 권한, 지침, 스킬, usage, 업데이트, 진단이 모두 의미 있는 기능이지만, 처음 들어온 사용자는 무엇을 먼저 봐야 할지 흔들린다.

## 가장 큰 리스크 5개

1. 다른 PC 등록 실패 시 원인 분류가 약하다.

   Tailscale 미설치, 다른 tailnet, firewall, stale connector, Bun/Git 문제, token mismatch가 사용자의 눈에는 모두 "등록 안 됨"으로 보일 수 있다.

2. 재부팅 후 신뢰가 부족하다.

   login task가 설치되어도 실제 재로그인/재부팅 후 server와 connector가 항상 복구되는지 충분히 증명되지 않았다.

3. 업데이트가 실패했을 때 복구 경로가 부족하다.

   업데이트 진행/성공/실패 표시는 생겼지만, 실패 후 자동 rollback, 재시도, 다음 부팅 처리까지는 약하다.

4. end-to-end 실제 Claude run 검증이 부족하다.

   연결됨 상태와 실제 Claude streaming, approval, image preview, 파일 접근은 별개의 신뢰 문제다.

5. 보안 사고 면적이 크다.

   원격으로 Claude Code를 실행하는 도구이기 때문에 workspace root, token, daemon port, command copy UX가 한 번만 잘못돼도 피해가 크다.

## 가장 먼저 해야 할 개선

1. `doctor`를 제품의 중심 기능으로 만든다.

   서버, registry, selected device, local daemon, advertised daemon URL, token match, behavior, Claude CLI, workspace root, login task를 하나의 JSON으로 검사하고 UI에 그대로 보여줘야 한다.

2. 다른 PC 등록 스크립트를 reconcile 스크립트로 격상한다.

   "설치한다"가 아니라 "현재 상태를 판별하고 보정하고, 마지막에 서버에서 실제로 접근 가능한지 검증한다"가 되어야 한다.

3. 실제 Windows PC 테스트 매트릭스를 만든다.

   깨끗한 PC, 기존 dirty repo, Bun 없음, Git 없음, Tailscale 없음, firewall 차단, port 점유, stale token, 재부팅 후 복구를 케이스로 둔다.

4. UI 첫 화면을 상태 판정 중심으로 더 밀어붙인다.

   설명보다 "이 브라우저/이 서버/이 디바이스가 지금 사용 가능한가"를 먼저 보여줘야 한다.

5. 보안 경고를 액션 가까이에 붙인다.

   token이 들어간 명령, connector port, workspace root unrestricted 옵션, device removal/uninstall에는 더 명확한 위험 표시가 필요하다.

## 통과 기준

DeskRelay를 파워유저용 개발도구로 "믿고 권할 수 있음"이라고 말하려면 최소한 아래 기준은 통과해야 한다.

- 새 Windows 사용자 프로필에서 서버 설치가 한 번에 성공한다.
- 다른 Windows PC에서 등록 명령을 세 번 반복 실행해도 같은 device row로 수렴한다.
- Tailscale 없음, firewall 차단, port 점유, stale token을 각각 다른 메시지로 분류한다.
- 재부팅 후 서버 PC와 등록 PC가 자동으로 online으로 복귀한다.
- 실제 Claude Code `ping`, 긴 streaming, approval, 이미지 첨부, 생성 이미지 preview가 새로고침 없이 동작한다.
- 업데이트 실패 시 사용자가 다음 행동을 바로 알 수 있다.
- `doctor` 출력만 보고 어느 노드가 실패했는지 판단할 수 있다.

## 최종 평가

DeskRelay는 좋은 아이디어를 가진 약한 앱이 아니라, 강한 아이디어와 강한 기능을 가진 거친 개발도구다. 지금의 목표가 출시가 아니라 파워유저용 self-host 도구라면 방향은 맞다.

다만 냉정히 말하면 현재 점수는 기능이 아니라 신뢰에서 깎인다. 설치, 연결, 재부팅, 업데이트, 진단이 단단해지기 전까지는 "멋진 도구"이지 "믿고 맡기는 도구"는 아니다.
